# 境界づけられたコンテキストの分析

## 1. 制約・スコープ

- MVP / プロトタイプか、それとも長期的なプロダクトか？
- 市場投入までの時間がどの程度猶予があるか？その時期は見直すことが可能か？
- リビルド・リアーキテクチャを行う理由は？ ROIの観点から判断する
- 外部起因で考慮する制約はあるか？

## 2. 責任分界点

- 概念モデルが整理されていて、既存の問題領域 (ドメイン) とどう対応するか
- ドメインモデル・データモデルがある場合は、解決領域 (コンテキスト) とどう対応するか
- 関連するサービスとの関係性は？ (コンテキストマップ)
- 類似するコンテキストを探す。車輪の再発明になっていないか。プロダクト全体としての整合が取れているか。
- マーケットに存在するSaaSやPaaSで代替できるものがあるか (作らないことが最良な選択な場合が多い)
- 可用性に影響を与えるサービスはあるか？
- 異常が発生した場合のリカバリー戦略は？サービスを横断するシステム障害に備えているか？
- その他、分割統治(統合・分割)の根拠と論拠は？

## 3. 運用チーム

- どのチームで運用する想定か？ (システム運用とプロダクト運用)
- 関連するチーム・組織との関係性は？ (組織パターン・コミュニケーションプロトコル)
- チームの規模と専門性はどの程度か？問題領域の複雑性を加味して適切なアサインか？
- デプロイが独立して行えるか？
- 技術スタックの想定は？社内標準を満たす内容か？標準外を選択する場合の理由は？
